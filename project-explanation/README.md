## 1. Executive Summary & Scope

**MusePath** is a modern, high-contrast web application designed for self-directed musicians who want personalized, AI-generated practice roadmaps. Standard music planning apps rely on static, linear courses. MusePath utilizes **Google Gemini Generative AI** to output structured roadmaps custom-tailored to the user's instrument, experience level, interest genres, practice time, and learning mood.

### Features:
* **Custom AI Roadmaps**: Multi-month, multi-week plans generated on-demand.
* **Active Plan Switching**: Support for creating, listing, and switching between multiple custom instrument roadmaps.
* **3D Terrain Visualizer**: A custom canvas visualizer rendering real-time 3D projected audio grids.
* **Metal Distortion Synth**: Interactive electric guitar synth using oscillator detuning, waveshaper distortion curves, and low-pass cabinet filters.
* **Device Responsiveness**: Full layout adjustments including hamburger headers, side overlays, and typography clamp scaling for standard devices.

## 2. System Architecture & Data Flow

MusePath is built as a decoupled Client-Server architecture:
* **Frontend**: React.js SPA initialized using Vite. States are managed using a lightweight **Zustand** store. Lucide Icons are used for visual indicators.
* **Backend**: Node.js web server running Express.
* **Database & Auth**: Hosted PostgreSQL database on **Supabase** with integrated email authentication.
* **AI API**: Google Generative AI Node SDK (`gemini-2.5-flash` model).
* **Metadata APIs**: Spotify search endpoints and YouTube tutorial lookups.

## 3. Database Design & Schema

The database uses a PostgreSQL schema on Supabase. Row-level security (RLS) policies are active, and an database trigger automatically creates user profiles.

### Core Tables:

1. **`users`**
   * `id`: UUID (PK, references auth.users)
   * `email`: Text
   * `streak_days`: Integer (current practice streak)
   * `xp`: Integer (gamified points)
   * `instrument`, `skill_level`, `goal_duration`: Varchar (stores active configuration parameters)

2. **`learning_plans`**
   * `id`: UUID (PK)
   * `user_id`: UUID (FK, references users.id)
   * `title`, `summary`: Text
   * `instrument`, `skill_level`, `goal_duration`: Varchar
   * `plan_data`: JSONB (contains full month/week layout details generated by Gemini)
   * `is_active`: Boolean (only one plan is active at a time)

3. **`learning_weeks`**
   * `id`: UUID (PK)
   * `plan_id`: UUID (FK, references learning_plans.id)
   * `user_id`: UUID (FK, references users.id)
   * `month_number`, `week_number`: Integer
   * `title`, `practice_goal`: Text
   * `topics`, `skills`, `youtube_searches`: Text[] / JSON
   * `is_completed`: Boolean (tracks checklist state)

4. **`progress`**
   * `id`: UUID (PK)
   * `user_id`, `plan_id`: Foreign Keys
   * `completed_weeks`, `total_weeks`: Integer
   * `completion_percentage`: Numeric

5. **`practice_logs`**
   * `id`: UUID (PK)
   * `user_id`, `plan_id`: Foreign Keys
   * `duration_minutes`: Integer
   * `notes`: Text
   * `practiced_at`: Timestamp (renders dashboard heatmap blocks)

## 4. Key Technical Implementations

### A. Asynchronous Background Enrichment (Speed Optimization)
* **Problem**: Statically querying the Spotify API for 48–96 songs during plan generation took 15+ seconds, hitting rate limits and causing browser timeouts.
* **Solution**: Decoupled the API flow. The server immediately returns the raw plan structure returned by Gemini, writes initial rows, and sends a fast success JSON response (`res.json`) to the user in **under 5 seconds**. Right after returning, a background task (`setTimeout`) queries Spotify for previews, IDs, and artwork concurrently, updating the `plan_data` database column asynchronously. Fallback icons are used until the record completes.

### B. Web Audio API Distorted Guitar Synth
* **Synth Architecture**: Custom Web Audio context:
  * Stacks multiple detuned sawtooth oscillators to create a thick chorus tone.
  * Connects oscillators to a `WaveShaperNode` utilizing a custom `Float32Array` distortion curve mapping (clipping shape).
  * Passes output through a low-pass `BiquadFilterNode` tuned to 2.5 kHz to simulate a real speaker cabinet frequency roll-off (eliminating harsh high-end harmonics).

### C. 3D Frequency Terrain Visualizer
* **Projection Logic**: Coordinates are created in a grid array representing 3D coordinates `(x, y, z)`. The height `z` is dynamically modified using frequency bins fetched from an audio `AnalyserNode`. Coordinates are projected into 2D canvas coordinates using standard perspective projection:
  `x_projected = center_x + (x * perspective_ratio)`.
  Transformation matrix equations rotationally tilt the mesh dynamically when a user hovers or moves the mouse.

### D. Stateful Mobile Sidebar Drawer
* **Responsive Styling**: A stateful header component (`mobile-header`) manages toggles for mobile viewport sizes. In the CSS, `.app-sidebar` is explicitly forced to `width: 280px !important;` under 768px media queries (since `--sidebar-width` is set to `0px` to let the page fill the screen, which would otherwise compress the sidebar to zero width).

## 5. Interview Prep & Core Q&A

### Q1: How did you implement distortion in the Web Audio context?
**Answer**: Distortion is created using a `WaveShaperNode` configured with a mathematical waveshaping curve. We populate a `Float32Array` (length 44100) using the distortion formula: 
`f(x) = (3 + k)x / (3 + k|x|)`
Where `k` is the distortion factor (e.g. 400). As the audio signal passes through, its wave peaks are clipped, producing high-frequency harmonics that sound like raw distortion. We low-pass filter the output below 2.5 kHz to simulate speaker cabinet roll-off.

### Q2: What is the benefit of using a Zustand store instead of React Context?
**Answer**: Zustand is a lightweight, external state management solution that runs outside the React render cycle. It uses a publish-subscribe pattern which prevents unnecessary re-renders of unrelated components. Unlike React Context (which triggers a full re-render of the subtree whenever any value in the context changes), Zustand allows components to selectively subscribe to small state slices, resulting in higher rendering performance.

### Q3: How do you handle database security and authorization?
**Answer**: We implement **Row Level Security (RLS)** in Supabase. Every database table has active policies ensuring that a user can only select, insert, update, or delete records where the `user_id` column matches their authenticated UUID (fetched securely from `auth.uid()`). This prevents data leakages across different users.
